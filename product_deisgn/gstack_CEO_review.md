## 一、最關鍵的 3 個戰略問題

### 問題 1：產品定位存在身份危機 — 個人工具 or 多用戶平台？

PRD §1.2 寫「主要用戶：跨國/跨幣別金融資產的個人投資者」，但同時提到「家庭成員共享」「理財顧問只讀模式」「多用戶平台」（§3.10, P2）。§8 競品對照表把所有未開發的功能標為 ✅，暗示這是一個與 Kubera ($249/yr) 競爭的產品。

**核心矛盾**：你是在做一個**給自己用的工具**，還是一個**給市場用的產品**？

這兩條路的架構、優先級、品質要求完全不同：

| 維度 | 個人工具 | 市場產品 |
|------|---------|---------|
| 認證 | 單用戶密碼/本機 | OAuth + 2FA + RBAC |
| 數據輸入 | 可以粗糙 | 需要防呆、導引、空狀態設計 |
| 多幣別 | 只做自己用的幣別 | 需覆蓋所有 ISO 4217 |
| 部署 | docker compose up | 安裝文檔、troubleshooting、社區支持 |
| 測試 | 手動驗證 | 單元 + E2E + CI |

**建議**：在 PRD §1 明確寫下產品定位。

---

### 問題 2：Notion MVP (Phase 1) 在驗證什麼？

Phase 1 預計花 2–4 週做 Notion MVP。但 Notion 的限制如此嚴重（無互動式 Pie Chart、無 TWR/MWR、公式能力有限），你的核心價值主張 —— **「深度分析和可視化工具」** —— 在 Phase 1 完全無法被驗證。

| Notion MVP 能驗證的 | Notion MVP 不能驗證的 |
|---|---|
| 數據模型是否合理 | 圖表互動體驗 |
| 需要哪些資產類別 | Dashboard 載入速度 |
| 匯率/報價更新流程 | 深色/淺色主題效果 |
| 你是否真的會持續錄入數據 | 用戶留存（只有你一個人） |

**三個選擇**：
1. **保留 Notion MVP**：但把目標明確為「驗證數據模型 + 養成錄入習慣」，2 週封頂，不做任何腳本自動化
2. **跳過 Notion，直接做最小 Web App**：一個頁面、一張表、手動 CRUD、一個 Pie Chart —— 同樣 2 週可完成，但直接在目標技術棧上驗證
3. **用 Google Sheets 替代 Notion**：公式能力更強、圖表更好、遷移更容易

### 問題 3：「全球 7 國 8 幣別」是真需求還是規格膨脹？

PRD 列出 US/HK/CN/JP/EU/UK/BR 七個地區。需要確認實際需求範圍。

**原因**：每多一個地區 = 多一種幣別 + 多一組數據源 + 多一套測試場景。7 個地區的排列組合是巨大的。

## 二、產品需求層面的 8 個缺口

### 缺口 1：缺少用戶旅程 (User Journey)

PRD 描述了功能，但沒有描述**日常使用場景**。

| 時間點 | 用戶在做什麼？ | PRD 是否覆蓋？ |
|--------|--------------|---------------|
| Day 1 | 初次設置 — 錄入所有現有持倉 | ❌ 無 onboarding 流程描述 |
| 每日 | 打開 Dashboard 看淨值變化 | ✅ 有 |
| 每週 | 記錄一筆新買入的股票 | ❌ 新增持倉的 UX 流程未描述 |
| 每月 | 對帳 — 核對 Dashboard vs 實際券商數據 | ❌ 完全未提及 |
| 每季 | 審視配置、考慮再平衡 | ⚠️ P1，但流程不清 |
| 年底 | 看年度績效、準備稅務 | ⚠️ P2，很遠 |

**建議**：在 PRD 新增「日常使用場景」章節。

### 缺口 2：數據輸入是產品成敗的核心，但著墨最少

所有 portfolio tracker 的第一大挑戰是**數據進來**。需要回答：

- 新增持倉時，哪些欄位必填 vs 選填？
- 輸入 Ticker 時是否有自動搜索/補全？
- CSV 格式因券商而異，你要支持哪些券商的格式？
- 買入均價如何計算？FIFO? LIFO? 加權平均？
- 如果 Ticker 找不到報價（如港股私募基金），怎麼辦？

### 缺口 3：空狀態 (Empty State) 設計

Dashboard 在**零數據**時長什麼樣？需要：
- 引導式 onboarding
- 範例數據模式

### 缺口 4：股票拆分、合併、除息等公司行動 (Corporate Actions)

PRD 完全未提及。但如果持有 NVDA 並發生 10:1 拆股，持倉數量和成本基礎需要自動調整。

### 缺口 5：手動估值資產的更新機制

P0 資產類別包含「銀行定期存款」。定期存款沒有 Ticker、沒有市場報價。Schema 中的 `last_price` 欄位對這類資產不適用。

### 缺口 6：報價更新頻率 vs 數據源成本的矛盾

FMP 免費層只有 250 次/天。需明確計算 API 用量預算。

### 缺口 7：衍生品 (P2) 的定義不清

「手動輸入市值」與「完整建模含 Greeks」之間差距巨大。

### 缺口 8：競品對照表 (§8) 存在誤導

PRD §8 把未實現的功能標為 ✅。應改為三態標記。

## 三、技術文檔層面的 4 個風險

### 風險 1：過早的架構決策

| 決策 | 問題 |
|------|------|
| Monorepo | 增加配置成本 |
| Redis cache | MVP 不需要 |
| Caddy reverse proxy | MVP 開發階段不需要 |
| tRPC 遷移路徑 | 過早規劃 |

### 風險 2：Schema 對「帳戶 vs 持倉」的耦合

`holdings.account_id REFERENCES accounts(id)` — 每個持倉必須屬於一個帳戶。但常見場景是非帳戶資產（硬體錢包、房產等）。

**建議**：`account_id` 應為 nullable。

### 風險 3：`NUMERIC(20,8)` 統一精度可能不夠

加密貨幣有 18 位小數。建議 quantity 改為 `NUMERIC(30, 18)`。

### 風險 4：MCP Server 在文檔中的權重與優先級不匹配

MCP 是 Phase 4 功能，但在文檔中大量描述，造成認知失調。
