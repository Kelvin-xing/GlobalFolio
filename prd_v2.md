# Globalfolio — 產品需求文檔 (PRD)

## 1. 產品概述

### 1.1 願景
打造一個全球化、多資產類別的個人財務投資組合管理 Dashboard，讓用戶在一個統一介面中管理和追蹤分散在全球各地的銀行存款、股票、債券、基金、加密貨幣等金融資產，並提供深度分析和可視化工具。

### 1.2 產品定位

本產品定位為**面向市場的多用戶平台**（非個人工具）。從 P0 起即以多用戶**架構**設計（Schema 含 user_id、認證系統完備），但完整的多用戶**功能**（邀請/共享/角色權限）在 P1 交付。包含完整的認證系統、防呆數據輸入、空狀態引導和生產級品質標準。

### 1.3 目標用戶
- **主要用戶**：擁有跨國/跨幣別金融資產的個人投資者
- **次要用戶**：家庭成員（共享檢視）、理財顧問（只讀模式）
- **目標市場**：全球 7 個地區（美/港/中/日/歐/英/巴西）均為潛在需求市場

### 1.4 核心問題
- 資產分散在美國、香港、中國大陸、日本、歐洲、英國、巴西等多國銀行/券商
- 缺乏統一視圖整合現金、股票、債券、基金、衍生品、保險等所有資產
- 多幣別換算和資產配置分析手動操作繁瑣
- 收益率、股息追蹤、稅務報表缺乏自動化

### 1.5 Priority 定義

| 概念 | 定義 | 說明 |
|------|------|------|
| **P0** | Must-have | 產品不能缺少此功能，否則無法向用戶交付價值。MVP 必須包含。 |
| **P1** | Should-have | 核心體驗的重要補充，MVP 後緊接著做。 |
| **P2** | Nice-to-have | 提升競爭力，但可延後。 |
| **P3+** | Future / 需單獨 PRD | 需獨立評估，不在當前路線圖中承諾。 |

## 3. 功能需求

### 3.1 Dashboard 主頁 (P0)

> **用戶故事**：作為一個跨國投資者，我希望打開 Dashboard 後一眼看到我的淨值和各類資產配置，這樣我能快速了解整體財務狀況而無需登入多個銀行/券商網站。
>
> **驗收標準**：首屏顯示淨值卡片（含與前一日漲跌幅）+ 2 個 Pie Chart + 持倉列表前 10 項，數據載入 < 3 秒。首屏視覺優先級：① Pie Chart（資產配置一目了然）② 淨值數字 ③ 與前一日漲跌幅/百分比。

- **淨值總覽卡片**：總資產、總負債、淨值，**與前一日漲跌幅及百分比**（首屏核心數據）
- **Pie Charts 區域**（參考 M1 Finance "Pie" 概念）：
  - **P0**：按資產大類分配 (存款/股票/債券/基金/加密) + 按幣別分配 (USD/HKD/CNY/JPY/EUR/GBP/BRL)
  - **P1**：按風險等級分配 (高/中/低風險) + 按帳戶/機構分配
- **持倉列表**：各資產明細，顯示名稱、數量、市值、漲跌幅、佔比
- **排序/篩選**：按市值、漲跌幅、資產類別排序

### 3.2 淨值歷史走勢 (P0)

> **用戶故事**：作為投資者，我希望看到淨值隨時間的變化趨勢，這樣我能評估投資策略是否有效。
>
> **驗收標準**：折線圖正確渲染至少 1 年的歷史數據，支持 3 種以上時間範圍切換。

- 折線圖顯示總淨值隨時間變化
- 時間範圍選擇：1D / 1W / 1M / 3M / 6M / YTD / 1Y / 5Y / ALL
- 可切換顯示各資產類別的走勢疊加

### 3.3 資產配置分析 (P1)

> **用戶故事**：作為投資者，我希望對比當前配置和目標配置的偏差，這樣我知道需要再平衡哪些資產。

- 當前配置 vs 目標配置對比
- 再平衡建議
- Treemap / Sunburst 多層次配置視圖

### 3.4 收益率與報酬率 (P1)
- 持有期報酬率 (HPR)
- 年化報酬率 (CAGR)
- 時間加權報酬率 (TWR) vs 資金加權報酬率 (MWR)
- 與基準指數對比 (S&P 500、MSCI World 等)

### 3.5 股息/利息追蹤 (P1)
- 股息日曆
- 年度股息收入統計
- 殖利率分析
- 存款利息追蹤

### 3.6 多幣別支援 (P0)

> **用戶故事**：作為持有 7 國資產的投資者，我希望所有資產自動按我選擇的基準幣別匯總展示，這樣我能在不同幣別間準確比較資產價值。
>
> **驗收標準**：用戶在設定頁選擇 base currency 後，所有金額同時顯示原始幣別和轉換值，匯率來源標示「Last updated」時間戳。

- **基準幣別 (Base Currency)**：用戶在設定中選擇基準幣別（如 USD），所有匯總金額以此幣別顯示
- 即時匯率轉換（每日更新）
- 顯示原始幣別 + 轉換後基準幣別
- 匯率波動對資產影響的分析
- 支援幣別：USD, TWD, HKD, CNY, JPY, EUR, GBP, BRL 等
- 台灣市場數據源：TWSE API 或 Yahoo Finance（`.TW` 後綴）

### 3.7 交易記錄 (P1)
- 買入/賣出/轉帳/股息記錄
- 交易歷史搜索與篩選
- CSV/Excel 匯入匯出

### 3.8 目標設定 & 進度追蹤 (P2)
- 退休金目標、買房目標等
- 進度可視化
- 預測到達時間

### 3.9 稅務報表 (P2)
- 已實現損益統計
- 資本利得稅估算
- 按國家/地區的稅務規則

### 3.10 多用戶 & 權限 (P1)
- 家庭成員帳戶分離/合併檢視
- 理財顧問只讀模式
- 角色權限管理（Admin / Member / Viewer）
- OAuth 2.0 + 2FA (TOTP) 認證

### 3.12 數據輸入詳細設計 (P0)

> 數據輸入是 portfolio tracker 的核心體驗。參考 Sharesight（3 種入口 + 自動價格填充）和 Ghostfolio（活動為中心的模型）。

#### 3.12.1 手動新增持倉

| 欄位 | 必填/選填 | 說明 |
|------|----------|------|
| 帳戶 | 必填 | 從已建立帳戶列表選擇，或「+ 新建帳戶」 |
| 資產類別 | 必填 | 股票/ETF/債券/基金/加密/現金/其他 |
| Ticker / 名稱 | 必填 | Ticker 自動搜索補全（連接 FMP/Yahoo API），找不到時回退到手動輸入名稱 |
| 數量 | 必填 | 支持小數（加密貨幣 18 位） |
| 買入均價 | 必填 | 自動帶入當前市價作為預設值 |
| 幣別 | 必填 | 自動從 Ticker 推導，用戶可覆寫 |
| 買入日期 | 選填 | 預設今日 |
| 費用 | 選填 | 手續費/佣金 |
| 備註 | 選填 | 自由文字 |

#### 3.12.2 手動估值資產

無 Ticker 的資產（定期存款、保險、房產等）使用不同模型：

| 欄位 | 必填/選填 | 說明 |
|------|----------|------|
| 帳戶 | 選填 | 可為 null（如「硬體錢包」「房產」不屬於任何機構帳戶）— 與 §3.12.1 不同，此類資產帳戶非必填 |
| 資產名稱 | 必填 | 自由文字 |
| 資產類別 | 必填 | 現金/保險/房產/其他 |
| 當前價值 | 必填 | 用戶手動輸入 |
| 估值方式 | 必填 | `manual`（手動更新）/ `formula`（如定期存款 = 本金 × (1 + 年利率 × 天數/365)） |
| 幣別 | 必填 | |
| 上次更新時間 | 自動 | 系統記錄，Dashboard 上顯示「N 天前更新」警示 |

#### 3.12.3 CSV 匯入 (P1)

- **欄位映射介面**：用戶上傳 CSV 後，系統顯示前 5 行預覽，用戶拖拽映射欄位（日期 → date、數量 → quantity 等）
- **重複檢測**：相同日期 + Ticker + 數量 + 金額視為重複，提示用戶確認
- **錯誤處理**：逐行驗證，錯誤行標紅並允許修正後重試
- **支援格式**：CSV, Excel (.xlsx)
- **券商預設模板**（P2）：提供常見券商（Interactive Brokers、富途、Schwab）的 CSV 格式預設映射

#### 3.12.4 成本基礎計算

- **預設方法**：加權平均成本法（Average Cost）— 最通用，適合多數用戶 (P0)
- **P2 擴展**：支持 FIFO（先進先出）— 美國稅務報告需要
- **Tax Lot 追蹤**（P2）：記錄每筆買入的獨立成本和日期，用於精確稅務計算
- **計算觸發**：每次新增買入/賣出交易時自動重算持倉成本基礎

### 3.14 公司行動 (Corporate Actions) (P1)

> 參考 Sharesight 的公司行動處理（業界最佳實踐）。Ghostfolio 至今未支援 stock split，是其最大功能缺口之一。

#### 3.14.1 支援的公司行動類型

| 類型 | 說明 | 對持倉的影響 | 優先級 |
|------|------|------------|--------|
| **Stock Split** | 拆股（如 NVDA 10:1） | 數量 ×10，成本基礎 ÷10 | P1 |
| **Reverse Split** | 合股（如 GE 1:8） | 數量 ÷8，成本基礎 ×8 | P1 |
| **Dividend (Cash)** | 現金股息 | 記錄收入，不改變持倉 | P1 |
| **Dividend (DRIP)** | 股息再投資 | 增加持倉數量，成本基礎為股息金額 | P1 |
| **Spin-off** | 拆分出新公司 | 新增子公司持倉，按比例分配原成本基礎 | P2 |
| **Merger/Acquisition** | 合併/收購 | 舊 Ticker 下架，轉換為新 Ticker + 現金（如有） | P2 |
| **Name/Ticker Change** | 名稱或代碼變更 | 更新 Ticker，持倉數據不變 | P2 |
| **Return of Capital** | 返還資本 | 降低成本基礎，不計為收入 | P2 |
| **Bonus Issue** | 紅股 | 增加數量，總成本基礎不變（單位成本攤薄） | P2 |

#### 3.14.2 實現方式

- **P1**：手動記錄 — 用戶在交易記錄中選擇「公司行動」類型，系統自動調整持倉數量和成本基礎
- **P2 迭代**：自動檢測 — 通過市場數據 API 偵測已持有股票的拆股/除息事件，推送通知讓用戶確認
- **交易記錄格式**：

```json
{
  "type": "CORPORATE_ACTION",
  "subtype": "STOCK_SPLIT",
  "holding_id": "uuid",
  "ratio_from": 1,     // 舊比例
  "ratio_to": 10,      // 新比例
  "effective_date": "2024-06-10",
  "note": "NVDA 10:1 forward split"
}
```

#### 3.14.3 數據模型

在 `transactions` 表中擴展 `type` 欄位：
- 新增類型：`split`, `reverse_split`, `drip`, `spinoff`, `merger`, `return_of_capital`, `bonus`
- 新增欄位：`ratio_from NUMERIC`, `ratio_to NUMERIC`（用於拆股/合股比例）

### 3.15 衍生品支援 (P3+ — 需單獨 PRD)

> 衍生品（期貨、期權、大宗商品）的完整建模涉及到期日、行權價、Greeks、保證金等複雜概念，與一般資產追蹤有本質區別。降級為 P3+ 並需要單獨 PRD 定義。

**暫行方案**（P0 即可用）：用戶可將衍生品作為「其他資產」手動輸入當前市值，僅做資產配置佔比展示，不做損益/Greeks 計算。

**P3+ 單獨 PRD 需覆蓋**：
- 期權定價模型（Black-Scholes / Binomial）
- 到期日管理與提醒
- 行權價追蹤
- Greeks 計算與展示（Delta, Gamma, Theta, Vega）
- 保證金追蹤
- 期貨合約展期 (Roll)

## 4. 數據輸入方式（按優先級）

### P0 — MVP 手動錄入
- **手動輸入**：通過 Web 表單逐筆新增持倉（見 §3.12.1）
- **手動估值**：無 Ticker 資產以手動市值方式記錄（見 §3.12.2）

### P1 — 數據匯入與核心功能
- **CSV/Excel 匯入**：批量匯入交易記錄，含欄位映射介面（見 §3.12.3）
- **公司行動手動記錄**：用戶手動記錄 stock split/dividend 等（見 §3.14）

### P2 — 數據自動化
- **Plaid API**：自動同步美國銀行/券商帳戶
- **券商 API**：Interactive Brokers、富途、老虎證券等
- **加密貨幣**：交易所 API (Binance, Coinbase) + 鏈上地址追蹤
- **公司行動自動偵測**：通過市場數據 API 偵測拆股/除息事件
- **Open Banking API**：歐洲 PSD2、英國 Open Banking、巴西 Open Finance
- **香港/日本券商**：自定義 API 整合或 CSV 匯入
- **中國大陸**：受限於監管，可能僅支援手動輸入

## 5. 數據安全與隱私

> 詳細安全架構和技術實現見 [tech_stack_v2.md](tech_stack_v2.md) §7。本節僅列出產品層面的安全原則。

### 5.1 架構原則
- **Self-hosted 優先**：敏感金融數據存儲在用戶自己的基礎設施上
- **加密**：數據庫加密 (AES-256 at rest)、TLS 1.3 in transit
- **無第三方數據銷售**：永不出售用戶數據
- **認證**：OAuth 2.0 + 2FA (TOTP) + RBAC

### 5.2 MCP (Model Context Protocol) 整合 (P3+)
- **作用**：MCP 可作為 AI 與金融數據的橋接層（P3+ 功能，見 §7 路線圖）
- **應用場景**：
  - Claude/ChatGPT 通過 MCP Server 安全存取投資組合數據
  - AI 驅動的資產配置建議
  - 自然語言查詢（"我的美股倉位佔比是多少？"）
  - 自動化交易記錄分類
- **Kubera 先例**：Kubera 已支援 MCP 整合，可做參考
- **安全考量**：MCP Server 需在用戶本地運行，不暴露到公網

## 9. 成功指標
